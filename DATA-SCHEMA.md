## File-Level Schemas

### `db/catalog.json`

**Purpose:** Master index of series and products (generated by `build_db.py`).

**Location in pipeline:** First stage; consumed by all downstream stages.

**Schema:**

```jsonc
{
  "series": [
    {
      "id": "K",                        // Series code (short)
      "name": "K-Series",               // Display name
      "folder": "K-Series",             // Asset folder name
      "description": "Korean-creative flagship. Curated heads, named personas, premium craft.",
      "character_count": 8,             // Count of characters in this series
      "body_count": 2                   // Count of body architectures
    },
    // … I, Fusion, SLE …
  ],
  "products": [
    {
      "code": "ZKE01_1+ZK155C",        // Unique product identifier
      "head_code": "ZKE01_1",           // Head/face sculpt
      "face_code": null,                // Optional face variant
      "body_code": "ZK155C",            // Body architecture
      "raw_body": "ZK155C",             // Parsed body code
      "series_id": "K",                 // Series it belongs to
      "folder_path": "K-Series/ZKE01_1+ZK155C",  // Asset folder
      "height_cm": 155,                 // Body height
      "cup_size": "C",                  // Bust cup size
      "body_line": "K-Series",          // Brand/line name
      "image_count": 18,                // Product photo count
      "video_count": 0,                 // Product video count
      "cover_image": "K-Series/ZKE01_1+ZK155C/ZKE01_1_ZK155C-101.jpg"
    },
    // … more products …
  ]
}
```

**Validation:**
- `series[].id` must be unique.
- `products[].code` must be unique.
- `products[].series_id` must reference a valid series.
- All asset paths must be relative to repository root.

### `db/body_profiles.json`

**Purpose:** Classification of body architectures by morphometric properties (WHR/BWR).

**Generated by:** `build_profiles.py` (reads `body_measurements.json` + `catalog.json`).

**Schema:**

```jsonc
{
  "families": [
    {
      "name": "The Classic",
      "description": "Timeless hourglass silhouette",
      "whr": [0.68, 0.72],              // [min, max] Waist-Hip Ratio
      "bwr": [1.4, 1.5],                // [min, max] Bust-Waist Ratio
      "silhouette": "Timeless hourglass",
      "premium": "+20%",                // Price tier adjustment
      "target": "First-time premium buyer",
      "color": "#FF6B9D"                // UI color (hex)
    },
    // … The Icon, The Muse, The Siren, The Empress, The Sculpt …
  ],
  "bodies": [
    {
      "body_code": "ZK155C",
      "series_id": "K",
      "height_cm": 155,
      "cup": "C",
      "weight_kg": 33.5,
      "bust": 85.2,
      "waist": 62.4,
      "hip": 91.3,
      "whr": 0.683,                     // Computed: waist / hip
      "bwr": 1.366,                     // Computed: bust / waist
      "family": "The Classic",          // Classified family (or null if out-of-range)
      "spec_card": "assets/Measure/ZK155C_pc_3.0.webp",  // Spec card image
      "notes": "Optional hand notes"
    },
    // … more bodies …
  ]
}
```

**Validation:**
- `whr` = `waist / hip` (must be in range [0.4, 0.9]).
- `bwr` = `bust / waist` (must be in range [1.0, 2.0]).
- `family` must be null OR match a family from `families[]`.
- `body_code` must match entries in `catalog.json`.

**Edge Cases:**
- **Out-of-range bodies:** When a body's WHR/BWR falls outside all family ranges, `family` is set to `null` (Unclassified).
- **Manual override:** The hand-curated `db/body_measurements.json` can force a `family` assignment, overriding the algorithm.

**Purpose:** Body architecture aggregates; denormalized for fast lookups.

**Generated by:** `build_characters.py`.

    "family": "The Muse",
    "silhouette": "Tall, hip-dominant",

**Status Distinction:**

| Field | `live` | `placeholder` |
|-------|--------|---------------|
| `photoshoot` | ✓ Real images exist | ✗ Null or absent |
| `placeholder` | ✗ Absent | ✓ SVG + direction |
| `story` | Always present | Always present (aspirational) |
| `price` | May be present | Null or absent |

**Validation:**
- `character_id` format: `{Series}-{BodyCode}-{Slot:02d}`.
- `slot` in [1, 2, 3, 4].
- Exactly 4 characters per body (no gaps in slots).
- `body_siblings` list must contain exactly 3 entries (the other 3 on the body).

### `db/character_overlay.json`

**Purpose:** Hand-curated persona edits and metadata overrides.

**Edit location:** Manual (non-generated).

**Schema:**

```jsonc
{
  "overrides": {
    "SLE-ZX160J-03": {
      "persona": {
        "name": "Vesper",               // Override if auto-generated name is wrong
        "title": "The Fantasy",         // Override title
        "tagline": "Pure imagination, made physical.",
        "story": "Custom narrative…",   // Hand-write full story
        "profile": {
          "personality": "…",
          "ideal_setting": "…",
          "signature": "…",
          "for_you_if": "…"
        }
      },
      "photoshoot": {
        "hero": "assets/SLE-Series/ZXE223_1+ZX160J/ZXE223_1+ZX160J-105.jpg"  // Override hero image
      }
    },
    // … more characters …
  },
  "notes": "Last updated: 2026-06-20. Edited names for K-Series, updated hero images for Fusion line."
}
```

**Merge strategy:**
- Overlay is **shallow-merged** into character records at build time.
- Only fields present in overlay override; others inherit from generated record.
- Missing overlay entries are skipped (no-op).

### `db/body_measurements.json`

**Purpose:** Hand-curated body specifications and measurement overrides.

**Edit location:** Manual (non-generated).

**Schema:**

```jsonc
{
  "bodies": [
    {
      "body_code": "ZK155C",
      "height_cm": 155,
      "cup": "C",
      "weight_kg": 33.5,
      "bust": 85.2,
      "waist": 62.4,
      "hip": 91.3,
      "bust_drop_cm": 22.8,             // Optional; calculated from bust/waist distance
      "spec_card": "assets/Measure/ZK155C_pc_3.0.webp",
      "family_override": "The Classic", // Force family assignment (overrides algorithm)
      "notes": "Verified measurements from product spec sheet v3.0"
    },
    // … more bodies …
  ]
}
```

**Merge strategy:**
- Measurements in this file are **authoritative** (override any computed values).
- `family_override` forces a family assignment (useful for edge cases).
- `notes` are for documentation only (not output).

**Input sources (per series):**

### `db/_stories_*.json`

**Purpose:** Story and profile input from writers (per series).

**Location:** `db/_stories_{series_slug}.json` (e.g., `_stories_k-series.json`).

**Schema:**

```jsonc
{
  "K-Series-ZK155C-01": {
    "story": "120–180 word narrative in second or third person. Grounded in real measurements. No torso references.",
    "profile": {
      "personality": "3–5 adjectives or a short phrase",
      "ideal_setting": "Where she belongs; the scene she evokes",
      "signature": "The one detail that makes her unmistakably hers",
      "for_you_if": "The buyer she's made for. Complete the sentence: 'For you if …'"
    }
  },
  // … more characters in this series …
}
```

**Validation:**
- `story` length: 120–180 words.
- Must reference actual body data (height, cup, family, silhouette).
- Must be grounded; no contradictions.
- `profile` fields are tight (phrase-length, not paragraphs).

**Output:** Merged into `db/character_stories.json` → then folded into `db/characters.json`.

### `db/.orchestrator/state.json`

**Purpose:** Build pipeline state tracking (idempotence, resume, diagnostics).

**Generated by:** `build_orchestrator.py`.

**Schema:**

```jsonc
{
  "timestamp": "2026-06-21T13:15:42.123Z",
  "duration_secs": 8.7,
  "status": "success",                 // "pending" | "running" | "success" | "partial_failure" | "failure"
  "exit_code": 0,                      // 0=success, 1=fatal, 2=skipped, 3=partial
  
  "stages": {
    "db": {
      "name": "db",
      "status": "complete",
      "start_time": 1718980542.123,
      "end_time": 1718980544.421,
      "duration_secs": 2.298,
      "input_hash": "sha256:abc123def456…",
      "output_hash": "sha256:xyz789…",
      "retry_count": 0,
      "max_retries": 3,
      "exit_code": 0
    },
    "profiles": {
      "status": "complete",
      "duration_secs": 1.654,
      "…": "…"
    },
    // … more stages …
  },
  
  "errors": [
    // Only populated if status = "partial_failure" or "failure"
    {
      "stage": "characters",
      "message": "Transient error: connection timeout",
      "timestamp": "2026-06-21T13:15:40Z",
      "attempt": 2
    }
  ],
  
  "notes": "Resumed from db stage; profiles + characters ran in parallel."
}
```

**Exit Codes:**
- **0 — Success:** All stages completed without error.
- **1 — Fatal Error:** Unrecoverable error; pipeline halted.
- **2 — Skipped:** No work to do (all inputs unchanged).
- **3 — Partial Failure:** Some stages failed; others succeeded. May be resumable.

**Input Hash Strategy:**
- Hash is computed from modification times + file sizes of declared inputs.
- If hash unchanged, stage is skipped (idempotence).
- `--reset` forces hash mismatch → full rebuild.

## Database Schema (SQLite)

**Tables:**

### `series`

```sql
CREATE TABLE series (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  folder TEXT NOT NULL,
  description TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### `bodies`

```sql
CREATE TABLE bodies (
  body_code TEXT PRIMARY KEY,
  series_id TEXT NOT NULL,
  height_cm INTEGER NOT NULL,
  cup TEXT NOT NULL,
  weight_kg REAL NOT NULL,
  bust REAL NOT NULL,
  waist REAL NOT NULL,
  hip REAL NOT NULL,
  whr REAL NOT NULL,
  bwr REAL NOT NULL,
  family TEXT,
  spec_card TEXT,
  notes TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY(series_id) REFERENCES series(id)
);

CREATE INDEX idx_bodies_series ON bodies(series_id);
CREATE INDEX idx_bodies_family ON bodies(family);
```

### `products`

```sql
CREATE TABLE products (
  code TEXT PRIMARY KEY,
  head_code TEXT NOT NULL,
  face_code TEXT,
  body_code TEXT NOT NULL,
  series_id TEXT NOT NULL,
  folder_path TEXT NOT NULL,
  image_count INTEGER DEFAULT 0,
  video_count INTEGER DEFAULT 0,
  cover_image TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY(body_code) REFERENCES bodies(body_code),
  FOREIGN KEY(series_id) REFERENCES series(id)
);

CREATE INDEX idx_products_body ON products(body_code);
CREATE INDEX idx_products_series ON products(series_id);
CREATE INDEX idx_products_head ON products(head_code);
```

### `characters`

```sql
CREATE TABLE characters (
  character_id TEXT PRIMARY KEY,
  slot INTEGER NOT NULL,
  body_code TEXT NOT NULL,
  series TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'live',
  persona_json TEXT,        -- JSON blob for persona (name, title, tagline, etc.)
  photoshoot_json TEXT,     -- JSON blob for photoshoot (images, price, sku)
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  UNIQUE(body_code, slot),
  FOREIGN KEY(body_code) REFERENCES bodies(body_code)
);

CREATE INDEX idx_characters_body ON characters(body_code);
CREATE INDEX idx_characters_series ON characters(series);
CREATE INDEX idx_characters_status ON characters(status);
```

### `character_stories` (optional)

```sql
CREATE TABLE character_stories (
  character_id TEXT PRIMARY KEY,
  story TEXT NOT NULL,
  profile_json TEXT,        -- JSON blob for profile fields
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY(character_id) REFERENCES characters(character_id)
);
```

## Build State & Metadata

### Execution History

Every build run appends to `db/.orchestrator/state.json`, creating an audit trail:

```json
[
  { "timestamp": "2026-06-20T10:00:00Z", "status": "success", … },
  { "timestamp": "2026-06-20T14:30:00Z", "status": "partial_failure", … },
  { "timestamp": "2026-06-21T09:15:00Z", "status": "success", … }
]
```

### Resume Mechanism

If a stage fails:

1. User runs `python scripts/build_orchestrator.py --resume`.
2. Orchestrator reads last entry from `state.json`.
3. Identifies first failed stage.
4. Reruns from that stage onward, reusing output from earlier stages.
5. Updates `state.json` with new attempt.

### Idempotence

Stages are skipped if:
- Input file hashes match previous run.
- All output files exist and are newer than inputs.
- No `--reset` or `--force` flag.

## Validation Rules

### Body Code Format

```regex
^Z[A-Z]\d{2,3}[A-J]$
```

Example: `ZG170C`, `ZX160J`, `ZK155C`

- `Z` = ZELEX prefix
- `[A-Z]` = Series letter (K, I, G/Fusion, X/SLE)
- `\d{2,3}` = Height in cm (155–173)
- `[A-J]` = Cup size

### Character ID Format

```regex
^[A-Za-z0-9\-]+\-Z[A-Z]\d{2,3}[A-J]\-\d{2}$
```

Example: `SLE-ZX160J-03`, `K-Series-ZK155C-01`

- Series name or code
- Body code
- Slot number (01–04, zero-padded)

### Series ID Values

Canonical series:
- `K` or `K-Series` (Korean-creative)
- `I` or `Inspiration` (Western naturalism)
- `Fusion` or `F` (Movable-jaw realism)
- `SLE` (Wide spectrum)

### WHR/BWR Ranges

**Minimum coverage:**
- WHR must be in [0.4, 0.9].
- BWR must be in [1.0, 2.0].

**Family assignment:**
- If WHR/BWR fall outside all family ranges → `family = null` (Unclassified).
- Manual `family_override` in `body_measurements.json` can force assignment.

### Cup Size Enum

Valid values: `A`, `B`, `C`, `D`, `E`, `F`, `G`, `H`, `I`, `J`

### Photo Path Format

All image paths must be relative to repository root:

```
assets/{Series-Folder}/{HeadCode}+{BodyCode}/{FileName}
```

Example:
```
assets/SLE-Series/ZXE223_1+ZX160J/ZXE223_1+ZX160J-101.jpg
```

## Examples

### Example: Adding a New Body

1. **Add to `db/body_measurements.json`:**
   ```json
   {
     "body_code": "ZX170D",
     "height_cm": 170,
     "cup": "D",
     "weight_kg": 38.5,
     "bust": 92.0,
     "waist": 67.5,
     "hip": 99.0,
     "spec_card": "assets/Measure/ZX170D_pc_3.0.webp"
   }
   ```

2. **Run build pipeline:**
   ```bash
   python scripts/build_orchestrator.py
   ```

3. **Pipeline automatically:**
   - Reads `body_measurements.json`.
   - Computes WHR (67.5/99.0 = 0.682) and BWR (92.0/67.5 = 1.363).
   - Classifies into family (The Muse, based on ranges).
   - Looks for 4 photoshoots with this body code.
   - Generates 4 characters (or placeholders if photoshoots missing).
   - Outputs to `db/characters.json`.

### Example: Editing a Character Persona

1. **Edit `db/character_overlay.json`:**
   ```json
   {
     "overrides": {
       "SLE-ZX170D-02": {
         "persona": {
           "name": "Aurora",
           "title": "The Dawn",
           "tagline": "Light breaks through."
         }
       }
     }
   }
   ```

2. **Run build:**
   ```bash
   python scripts/build_orchestrator.py
   ```

3. **Pipeline merges:**
   - Reads auto-generated character.
   - Applies overlay edits.
   - Outputs updated `db/characters.json`.
